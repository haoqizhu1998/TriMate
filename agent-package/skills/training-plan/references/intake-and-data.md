# 输入、赛事与分层读取

## 一次只推进一个问题

每条回复最多只问一个问题。优先询问会改变计划可行性或安全性的字段，其余缺失信息采用保守默认并明确写出。

生成计划预览的最低条件：

1. 至少一个赛事的项目和日期。
2. 每周可训练总时长或可训练天数。

优先补充顺序：

1. 当前疼痛或医学风险信号。
2. 目标赛事日期和项目。
3. 每周可训练时间及硬性不可训练日。
4. 最近 4 周训练量与最长训练。
5. 目标成绩或完赛目标。
6. 近期成绩、VDOT、FTP、阈值配速、心率等能力数据。
7. 设备、场地、偏好和软限制。

## 分层读取

不得读取某用户全部历史数据。只在已经解析出的当前用户范围内分层读取：

- **最近 28 天明细**：训练、完成情况、疲劳、睡眠、疼痛和恢复，用于当前负荷和近期安排。
- **最近 90 天摘要**：周量、关键课、能力变化、持续中断和重复风险，用于趋势判断。
- **长期关键画像**：稳定目标、运动背景、伤病史摘要、常用设备、长期可训练时间、偏好和限制。
- **最新有效计划**：只读取当前 `active` 计划及必要的最近变更摘要，不加载全部旧版本。

数据不足时降低置信度，不扩张读取范围来填满上下文。没有稳定身份时只给即时预览，不写长期计划。

## 赛事模型

支持多主赛和多热身赛。每个赛事至少包含：

- 赛事 ID、日期、项目、距离/组别。
- 角色：主赛或热身赛。
- 目标类型和目标值。
- 优先级、是否可调整、备注。

用户可通过表单式信息或自由文本添加赛事。自由文本能明确理解时，先复述解析结果并请求一次确认；不必把用户拉进逐字段问答。

新增、删除或修改任一赛事与目标后，都要重新评估计划。赛事过密或目标冲突时，推荐选择主赛或调整目标，但最终保留用户决定。

## 目标解释与默认排序

推荐的目标层级从稳健到激进为：

1. 根据计划推荐成绩。
2. 健康完赛。
3. 主成绩目标。
4. 过程目标。
5. 名次目标。

用户未标记时，按以上顺序推断，并在预览中说明。第三类或其他自由表达目标由系统理解后，向用户确认一次。

## 训练条件

把条件分成硬限制和软偏好：

- 硬限制：不可训练日期、每次最长时长、场地不可用、医学限制、必须休息日。
- 软偏好：喜欢早晨训练、希望周末长课、偏好室内骑行、讨厌某类训练。

同时记录设备与资源：功率计、心率带、跑步机、骑行台、泳池、开放水域、力量器械和可用地形。没有设备时切换到时长、RPE 或谈话测试，不因此卡住。

## 计划版本状态

- `preview`：新计划或重大改版的预览，未确认，不覆盖执行中计划。
- `active`：用户确认后唯一生效的当前计划。
- `paused`：用户暂时不能执行，但计划历史和目标保留。
- `superseded`：被新版本替换的旧计划，只保留摘要供偏差分析。
- `completed`：最后目标赛事结束并完成复盘。

任何时候最多只有一个最新有效计划。重大变化必须先生成 `preview`，用户确认后再转为 `active`。
